## Improvement 1 — Split the 2,457-Line Monolith

The entire app lives in one JSX file. This is the single biggest maintainability issue. A new developer has nowhere to start except searching for the`═══` delimiters. Refactor into logical modules:

```
src/
├── components/
│   ├── SWMM5LegoBuilder.jsx    # Orchestrator only (~200 lines)
│   ├── GridEditor.jsx          # Canvas + cell rendering
│   ├── ElementPalette.jsx      # Left panel
│   ├── StormSelector.jsx       # Right panel
│   ├── SwmmInspector.jsx       # Click-to-inspect panel
│   ├── ResultCharts.jsx        # Tabbed charts
│   ├── Toolbar.jsx             # Action buttons
│   ├── TutorialOverlay.jsx     # Onboarding
│   └── ExportModal.jsx         # INP preview/download
├── lib/
│   ├── elements.js             # EL{}, CATS[] — element definitions
│   ├── hydraulics.js           # cnInfiltration, manningPipe, buildModel, runSWMM5
│   ├── storms.js               # _F, _CD, _IF, _S, STORM_CATS, STORMS[]
│   ├── validation.js           # validateModel, autoFix
│   ├── exportInp.js            # exportINP()
│   ├── importInp.js            # importINP()
│   └── demos.js                # DEMOS[] array
└── hooks/
    ├── useSimulation.js        # Simulation state + animation loop
    ├── useGrid.js              # Grid state + undo history
    └── useInspector.js         # Cell inspection state
```

The`hydraulics.js` and`storms.js` modules become independently testable pure functions. This is the prerequisite for every other improvement.

## Improvement 3 — Dynamic Wave Routing

The kinematic wave approximation is the most significant hydraulic limitation. It cannot model backwater, surcharge, or reverse flow — all of which are common in the dense networks that users build on the grid.

The upgrade path from kinematic to dynamic wave requires solving the Saint-Venant equations:

$$
\frac{\partial A}{\partial t} + \frac{\partial Q}{\partial x} = 0
$$

$$
\frac{\partial Q}{\partial t} + \frac{\partial}{\partial x}\left(\frac{Q^2}{A}\right) + gA\frac{\partial y}{\partial x} + gA(S_f - S_0) = 0
$$

For a browser implementation, use the Preissmann slot method (same as EPA SWMM5) with a simplified two-point implicit scheme:

```js
// lib/hydraulics.js — add dynamic wave conduit step
function dynamicWaveStep(conduit, upNode, downNode, dt) {
  const { length, diameter, roughness } = conduit;
  const dh = (upNode.head) - (downNode.head);
  const slope = dh / length;

  // Preissmann slot: keep pipe full with virtual slot
  const slotWidth = 0.001 * diameter; // narrow slot
  const area = pipeArea(conduit.depth, diameter);
  const wetPerim = pipeWetPerimeter(conduit.depth, diameter);
  const R = area / wetPerim;

  // Friction slope (Manning's)
  const Sf = (roughness * conduit.flow) ** 2 / (area ** 2 * R ** (4/3));

  // Momentum equation (explicit step)
  const dQ_dt = -9.81 * area * (Sf - slope);
  conduit.flow += dQ_dt * dt;

  // Continuity: update depths at nodes
  return conduit.flow;
}
```

Even a partial implementation — adding the inertia term to the current kinematic solver — noticeably improves results for flat networks and surcharging conditions, which are exactly the scenarios users build in the LEGO grid.

## Improvement 4 — Per-Cell Property Editing

Every cell currently uses the global default properties from`EL{}`. A user who places a grass cell gets CN=39 everywhere. In real stormwater modeling, the same surface type has different properties depending on soil, slope, and condition. Add a right-click context menu for property overrides:

```jsx
// Context menu on right-click of any placed cell
function CellContextMenu({ cell, element, onUpdate, onClose }) {
  const defaults = EL[element];
  const [cn, setCn] = useState(cell.cn ?? defaults.CN);
  const [imperv, setImperv] = useState(cell.pI ?? defaults.pI);

  return (
    <div className="cell-context-menu" style={{
      position: 'fixed',
      background: '#1a1a2e',
      border: '2px solid #F2C717',
      borderRadius: 6,
      padding: 12,
      zIndex: 1000,
      minWidth: 200,
    }}>
      <div style={{ color: '#F2C717', fontFamily: 'Fredoka One',
                    marginBottom: 8 }}>
        Override: {defaults.label}
      </div>
      {defaults.CN !== undefined && (
        <label style={{ display: 'flex', justifyContent: 'space-between',
                        color: '#ccc', fontSize: 12 }}>
          Curve Number
          <input type="number" min={40} max={100} value={cn}
                 onChange={e => setCn(Number(e.target.value))}
                 style={{ width: 50, textAlign: 'right' }} />
        </label>
      )}
      <button onClick={() => { onUpdate({ cn, imperv }); onClose(); }}>
        Apply
      </button>
    </div>
  );
}
```

Store overrides in a parallel`cellProps` state:

```js

```

## Improvement 5 — CFL Validation Check

The current validation catches only structural errors (no outfall, disconnected links). It misses the most common cause of simulation instability: the Courant-Friedrichs-Lewy condition. Add it to`validateModel()`:

```js
// In validation.js — add CFL check
function checkCflConditions(model, dtRoute) {
  const warnings = [];

  model.conduits.forEach(conduit => {
    // Wave celerity for full pipe
    const celerity = Math.sqrt(9.81 * conduit.diameter);
    const courant = (celerity * dtRoute) / conduit.length;

    if (courant > 1.0) {
      warnings.push({
        type: 'cfl',
        conduit: conduit.id,
        courant: courant.toFixed(2),
        message: `Conduit ${conduit.id}: Courant = ${courant.toFixed(2)} > 1.0.
                  Pipe length ${conduit.length}ft may cause instability.
                  Consider splitting into ${Math.ceil(courant)} cells.`,
      });
    }
  });

  return warnings;
}
```

Display CFL warnings with an orange border on the affected pipe cells in the grid — same as the existing red border for errors.

## Improvement 6 — Time-Series CSV Export

After simulation, users cannot export the results. Every hydraulic engineer's next step after running a model is exporting time-series data for reports. Add a CSV download button to each results tab:

```js
// In lib/exportResults.js
export function exportResultsCsv(simResult) {
  const { systemHistory, timeLabels } = simResult;

  const rows = [
    ['Time (min)', 'Rainfall (in/hr)', 'Runoff (CFS)',
     'Pipe Flow (CFS)', 'Outfall Flow (CFS)'],
    ...timeLabels.map((t, i) => [
      t,
      systemHistory.rainfall[i]?.toFixed(3),
      systemHistory.runoff[i]?.toFixed(3),
      systemHistory.pipeFlow[i]?.toFixed(3),
      systemHistory.outfall[i]?.toFixed(3),
    ]),
  ];

  const csv = rows.map(r => r.join(',')).join('\n');
  const blob = new Blob([csv], { type: 'text/csv' });
  const url = URL.createObjectURL(blob);
  const a = document.createElement('a');
  a.href = url;
  a.download = `swmm5-lego-results-${Date.now()}.csv`;
  a.click();
  URL.revokeObjectURL(url);
}
```

Add equivalent exports for subcatchment, pipe, and node tabs.

## Improvement 7 — Save / Load to localStorage

Currently every browser refresh loses the model. Add persistence with a single`useEffect`:

```js
// In useGrid hook
const STORAGE_KEY = 'swmm5-lego-autosave';

// Save on every grid change
useEffect(() => {
  const saveData = {
    grid,
    gridSize,
    cellProps,
    stormIdx,
    savedAt: new Date().toISOString(),
  };
  localStorage.setItem(STORAGE_KEY, JSON.stringify(saveData));
}, [grid, gridSize, cellProps, stormIdx]);

// Restore on mount
useEffect(() => {
  try {
    const saved = localStorage.getItem(STORAGE_KEY);
    if (saved) {
      const { grid, gridSize, cellProps, stormIdx } = JSON.parse(saved);
      setGrid(grid);
      setGridSize(gridSize);
      setCellProps(cellProps ?? {});
      setStormIdx(stormIdx ?? 0);
    }
  } catch (e) {
    // Corrupted save — start fresh
    localStorage.removeItem(STORAGE_KEY);
  }
}, []);
```

Also add a **Named Saves** feature — let users save up to 5 named slots in localStorage for comparing different design scenarios.

## Improvement 9 — Expand Validation Rules

Current validation has 3 errors and 1 warning. Real SWMM5 models fail for many more reasons. Add:

```js
const ADDITIONAL_CHECKS = [
  {
    id: 'no_surface',
    check: (grid, model) => model.subcatchments.length === 0,
    severity: 'error',
    message: 'No surface elements — model will produce zero runoff',
  },
  {
    id: 'flat_network',
    check: (grid, model) => model.conduits.every(c => c.slope < 0.0001),
    severity: 'warning',
    message: 'All pipes are nearly flat — flow may stall or oscillate',
  },
  {
    id: 'isolated_node',
    check: (grid, model) => {
      const connected = new Set(model.conduits.flatMap(c => [c.from, c.to]));
      return model.nodes.some(n => n.type !== 'outfall' && !connected.has(n.id));
    },
    severity: 'warning',
    message: 'Node has no connected pipes and will receive no flow',
  },
  {
    id: 'storage_no_outlet',
    check: (grid, model) => {
      const storageIds = model.nodes
        .filter(n => n.type === 'storage')
        .map(n => n.id);
      const outletIds = new Set(model.conduits.map(c => c.from));
      return storageIds.some(id => !outletIds.has(id));
    },
    severity: 'warning',
    message: 'Storage node has no outlet pipe — basin will fill and overflow',
  },
];
```
